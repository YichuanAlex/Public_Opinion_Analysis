# 小红书帖子加热判断工具

这是一个本地工具，用来根据历史投放数据学习“什么样的帖子值得加热”，再对新帖子给出投前判断。

## 分享给同事使用

1. **打包文件夹**：把整个文件夹压缩成 ZIP 发给同事（微信 / 飞书 / 邮件都行）。
   - 如果想让大家用同一份历史数据，把那份 CSV / TSV 文件也一起放进去。
   - `cookies.txt`（个人登录态）和 `llm_config.json`（含 API Key）**不要一起发**，每人自己配置（或不配置）。

2. **同事收到后**：解压，双击 `start_tool.command` 即可。
   - 首次运行 macOS 可能提示"无法打开来自未知开发者的文件"→ 右键点文件 → 选"打开" → 再点"打开"。
   - 如果弹出提示说"未检测到 Node.js"，去 [nodejs.org](https://nodejs.org) 下载安装 LTS 版本（首页大绿按钮），装完再双击。

3. **历史数据各自管各自**：学习模型保存在本地浏览器（localStorage），每人的判断口径可以独立设置，互不影响。

---

## 推荐使用方式：跑本地后端

只用浏览器打开 `index.html` 也能用，但只能手动输入标题/正文，并且要自己上传封面图。
推荐启动本地 Node.js 后端，这样可以直接贴小红书笔记链接，自动抓取标题、正文、封面图和当前互动数据。

```bash
cd "只能判断帖子内容是否值得加热"
node server.js
```

启动后访问 [http://localhost:5173](http://localhost:5173)。

需要 Node.js 18+。`server.js` 只使用内置模块，无需 `npm install`。

工作流：

1. 从 Excel、飞书或金山表格复制包含表头的历史数据，粘贴到左侧文本框。
2. 点击“学习历史数据”。
3. 在右侧第一行粘贴小红书笔记链接（App 复制的 `http://xhslink.com/...` 或网页端 `https://www.xiaohongshu.com/explore/...` 都可以），点击“自动获取”。标题、正文、当前点赞/收藏/评论/分享、笔记 ID 和封面图都会自动填好，封面也会自动做视觉评分。
4. 检查抓回来的字段，需要的话手动修正。
5. 点击“判断”，查看是否值得加热、预测 CPE、建议测试金额和相似历史样本。

如果要使用 LLM 混合预测：

1. 启动本地后端后，在左侧“LLM 设置”里选择服务商并粘贴 API Key，或启动前设置环境变量：

```bash
OPENAI_API_KEY='sk-...' node server.js
```

2. 支持这些服务商：

- `openai`：默认走 OpenAI Responses API，支持传封面图。
- `deepseek`：走 OpenAI 兼容 Chat Completions，默认模型 `deepseek-chat`，默认不传封面图，只使用本地封面评分。
- `gemini`：走 Gemini `generateContent`，默认模型 `gemini-2.5-flash`，支持传封面图。
- `compatible`：适合 OpenRouter、SiliconFlow、阿里百炼、Qwen、火山等 OpenAI 兼容接口，需要填写 Base URL。

3. 也可以用环境变量指定：

```bash
LLM_PROVIDER='deepseek' OPENAI_MODEL='deepseek-chat' OPENAI_API_KEY='sk-...' node server.js
LLM_PROVIDER='gemini' OPENAI_MODEL='gemini-2.5-flash' OPENAI_API_KEY='...' node server.js
LLM_PROVIDER='compatible' OPENAI_BASE_URL='https://api.siliconflow.cn/v1' OPENAI_MODEL='Qwen/Qwen2.5-72B-Instruct' OPENAI_API_KEY='...' node server.js
```

4. 先点击普通“判断”，或直接点击“LLM混合判断”。工具会先用本地模型找相似历史样本和预测 CPE，再把新帖、封面、相似历史和本地预测交给 LLM 做二次校准。

页面里粘贴的 API Key 只保存在当前 `server.js` 进程内存，不写入文件；关闭本地服务后会失效。不要把自己的 API Key 分享给同事。纯文本模型建议关闭“支持时发送封面图片”，工具仍会把本地封面评分、亮度、清晰度等指标发给模型。

如果某些笔记被风控（页面返回登录墙或拦截页），后端会返回错误提示，这时仍可以退回到手动输入和手动上传封面。

## 关于 xsec_token 过期 / 抓不到的链接

小红书网页版的链接里有一段 `xsec_token=...`，这是分享时绑定到分享者会话的签名 token，通常几小时后就会失效。失效后访问，小红书会把请求重定向到 `/404?errorCode=-510001` 的错误页（页面里只有小红书 logo 和首页文案，没有真正的笔记内容）。

工具已经能识别这种 404 情况并报错，避免误把 logo 当作封面。两种解决思路：

### 1. 重新复制链接（最简单）

直接在小红书 App 里点该笔记，重新"复制链接"，token 就是新的，几分钟内贴进工具一般都能抓到。

### 2. 配置登录态 Cookie（一劳永逸）

如果想用浏览器里历史攒下来的链接，或者想稳定抓那些 token 已经过期的笔记，可以让 server 带上你的登录态 cookie：

1. 用浏览器登录 https://www.xiaohongshu.com
2. 打开开发者工具 → Network → 刷新页面 → 点任一请求 → 复制 `Cookie` 请求头的完整值（一长串 `a1=...; webId=...; web_session=...` 这样的字符串）
3. 在项目根目录新建一个 `cookies.txt`，把 cookie 字符串粘进去，保存
4. 重启 `node server.js`，控制台会输出 `已加载 cookie（长度 ...）`

或者用环境变量：

```bash
XHS_COOKIE='a1=...; webId=...; web_session=...' node server.js
```

注意：`cookies.txt` 里是你的登录态，不要提交到 git，也不要分享给别人。

也可以上传 `.csv`、`.tsv` 或 `.txt` 文件。不支持直接读取 `.xlsx`，如果数据在 Excel 里，可以复制粘贴整张表，或另存为 CSV/TSV。

## 需要的历史字段

工具会自动识别这些中文表头：

- 发布日期
- 投放截止
- 状态
- 链接
- 笔记ID
- 投放记录
- 标题
- 正文 / 内容
- 封面评分、封面亮度、封面对比度、封面饱和度、封面清晰度、封面信息密度
- 现有互动量 / 投前互动量
- 当前点赞、当前收藏、当前评论、当前分享
- 投后互动量
- 投放后点赞、投放后收藏、投放后评论、投放后分享
- 总互动量
- 总投放金额
- 综合 CPE

最关键的是“标题或正文”“投前/现有互动量”和“综合 CPE”。如果没有综合 CPE，但有“总投放金额”和“总互动量”，工具会自动计算 CPE。发布日期可以保留在表里作为记录字段，但不会参与判断。封面相关列不是必填；没有历史封面列时，工具会对新上传封面做本地视觉评分，并作为预测修正项。

注意：投放后的点赞、收藏、评论、分享是结果数据，不会作为投前判断信号使用，避免模型提前看到投放结果。

## 判断逻辑

默认把历史样本中 CPE 表现较好的部分视为“值得加热”，同时参考总互动规模。模型会学习：

- 标题关键词和标题形态
- 正文关键词、场景表达、痛点表达和行动引导
- 封面亮度、对比度、色彩、清晰度、信息密度和疑似人物元素
- 现有互动量区间
- 点赞、收藏、评论、分享的互动结构
- 相似历史样本的 CPE 和成功率

左侧可以切换为手动 CPE 阈值，也可以设置最低总互动量和保守程度。

## 注意

历史样本越多判断越准，适合做投前初筛和小额测试建议，不适合当作绝对结论。建议始终先小额跑 12-24 小时，再根据真实 CPE 决定是否加码。

## 持续学习：每月追加新加热数据

模型不是"在线进化"的，它每次导入数据时从零重训。所以"持续学习"=保持训练集最新。左侧导入面板里的「**＋ 追加本月新加热数据**」就是为这个设计的，不用手工合并表格：

1. 在金山/飞书把当月「加热汇总」tab 导出为 CSV（保持你原始的列：发布日期、状态、达人名称、链接、标题、投前互动量、各项投放后数据、总互动量、总投放金额、综合 cpe、一二级分类等）。
2. 点开「追加本月新加热数据」，上传或粘贴这份 CSV。
3. 点「追加并重训」。工具会自动：
   - 按**发布日期**补「加热渠道」（2026-03-01 前=口碑通，03-01~04-27=合作广场，04-28 起=薯条）；
   - 从**链接**提取「笔记ID」；
   - 按笔记ID/链接**去重**（同一条新数据覆盖旧的，方便修正 CPE）；
   - 并入现有训练集并**立即重训**，结果存到浏览器本地。
4. 完成后会显示「新增 X 条，更新 Y 条，去重后共 Z 条，渠道分布……」。

下个月重复一遍即可，训练集会越滚越大。导入的规范训练集也会保留在左侧大文本框里，可随时全选复制做备份。

## LLM 混合判断与 prompt 调优记录

工具在本地统计模型之上叠加了一层 LLM（DeepSeek / OpenAI / Gemini 可选）做综合判断。配置在左侧「LLM 设置」，会持久化到 `llm_config.json`（含 API Key，**分享时不要带上**）。

### 验证基准（2026-06-02）

用「口碑通 + 合作广场」共 573 条做训练集，「薯条」59 条做测试集（CPE ≤ 1.4 为成功硬线），跑了 `llm_validate.js`：

| 方案 | 准确率 | 精确率 | 召回率 | F1 | 误推差帖(FP) |
|------|--------|--------|--------|----|----|
| 本地模型 | 74.6% | 75.0% | 52.2% | 0.62 | 4 |
| LLM（旧 prompt） | 71.2% | 63.6% | 60.9% | 0.62 | 8 |
| **LLM（新 prompt）** | **74.6%** | **72.2%** | 56.5% | **0.63** | **5** |

### 调优结论

旧 prompt 的 LLM 容易掉进「内容质量陷阱」——把文案优美但主题宽泛（旅行随拍、风景、小确幸、宠物日常）的帖子误判为值得加热，而这类内容实际 CPE 往往很差。

新 prompt 的三处关键改动（见 `server.js` 的 `buildLlmPrompt` / `sanitizeLlmPayload`）：
1. **写死 CPE ≤ 1.4 的成功硬线**，决策严格按预测 CPE 区间映射。
2. **预先在后端算好相似历史的统计**（达标比例、CPE 中位数等）塞进 prompt，避免 LLM 自己做不可靠的算术。
3. **明确警告「内容质量是弱信号」**，强制以相似历史的真实 CPE 为主锚点。

效果：误推差帖从 8 降到 5，精确率回到 72%，F1 成为三者最高。

剩余误判多是「标题正能量、实际主题宽泛」的帖子，光看标题判不出，需要正文 + 封面图共同参与（DeepSeek 不支持视觉，换 gpt-4o / Gemini 可让 LLM 看封面）。

验证明细见 `llm_validation_results.csv`（旧 prompt）和 `llm_validation_results_v2.csv`（新 prompt）。
